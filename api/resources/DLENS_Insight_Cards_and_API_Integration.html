What Insight Cards are

Small, self-contained research notes that turn messy, fast-moving intel (videos, calls, blogs, filings) into:

  • a plain-English synthesis,
  • a machine-readable payload (JSON),
  • a verification section with primary sources, and
  • proposed score nudges (what to move, by how much, and why).

Each card lives as a single HTML file with an embedded JSON block:

<script type="application/json" id="dlens-card-json">{ ... }</script>

Why they’re valuable

  • Speed → judgment: Convert noise into decision-ready signals with explicit confidence.
  • Auditability: Every proposed DUU/Probability change cites source IDs + dates.
  • Repeatability: Same format every time; easy ingestion & rules application.
  • Guardrails: Only Verified claims move scores under v12; themes stay non-scoring until verified.
  • Portability: Human-friendly HTML, API-friendly JSON.

Standard sections (human)

  • Title, ID, tags
  • What’s Verified (primary sources: filings, press releases, operator remarks, official docs)
  • Claims with status: Verified / Partially Verified / Plausible (+ confidence)
  • Implications for DLENS (what to track, how to gate)
  • Proposed adjustments: [ticker, field, delta, reason]
  • Notes / expiry (6–18 months unless reconfirmed)

How they integrate with the API version

1) Ingest cards into your KB

  • Option A (HTML files): Upload the HTML; backend extracts the <script id="dlens-card-json"> block and stores it (e.g., /kb/cards.jsonl).
  • Option B (direct JSONL): Save the JSON payloads as newline-delimited JSON.
Suggested KB layout: /kb/cards.jsonl (append-only) and /kb/cards_index.json (IDs → metadata, tags, expiry).

2) Apply cards during report generation

Hunt/Spotlight services read the KB and apply only score-eligible adjustments.

  • GET /schemas/<name> → returns JSON schema for requests.
  • POST /spotlight/html → body includes ticker and card filters.
  • POST /hunt/html → same SSOT numbers; accepts card_ids and/or filters.

Example Spotlight request

{
  "ticker": "ASML",
  "horizon_years": 6,
  "as_of_utc": "2025-10-12T22:35:00Z",
  "card_filters": {
    "status_allowlist": [
      "Verified",
      "Partially Verified"
    ],
    "max_age_months": 18
  },
  "apply_rules": {
    "only_verified_moves_scores": true,
    "partially_verified_needs_dual_sources": true
  }
}

3) Keep cards fresh

  • Expiry rule: cards auto-expire after 6–18 months unless reconfirmed.
  • Promotion path: multiple aligned cards → create a Rule Addendum (versioned) so the model doesn’t re-evaluate the same premise.

4) SSOT consistency (Hunt ↔ Spotlight)

Hunt rows and Spotlights must share the same SSOT snapshot (inputs + applied card deltas). Your API should return an SSOT hash/ID with each artifact so QA can assert parity.

Operational best practices

  • Two-source policy for any market-moving claim.
  • Source registry: normalize citations to {publisher}:{date}:{slug} for deduping.
  • Confidence discipline: Avoid >0.95 unless filed or operator-owned materials.
  • Separate theme vs adjustment: only move scores when verification gates are met.

Developer TL;DR

  1. Parse JSON from each card’s <script id="dlens-card-json">.
  2. Store in /kb/cards.jsonl with an index.
  3. At report time: filter → validate gates → apply deltas → emit audit trail.
  4. Return HTML (Gold) + audit JSON so the front end can show “which cards moved what.”
